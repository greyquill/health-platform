FHIR R4

FHIR R4 (RESTful API)

FHIR is the modern healthcare data standard: clinical information is exposed as resources (Patient, Observation, Condition, and so on) over plain HTTPS, so any conformant system can read it. Greyquill ships a FHIR R4 RESTful API out of the box.

EndpointWhat it returns
GET /fhir/metadata The FHIR CapabilityStatement: a machine-readable description of exactly what this server supports, so an integrating system can discover it programmatically.
GET /fhir/Patient/:id A single Patient resource by id.
GET /fhir/Patient/:id/$everything A full patient-record export Bundle via the $everything operation: the whole record for that patient in one call.
GET /fhir/Observation?patient= An Observation searchset Bundle: vitals and results for the patient.
GET /fhir/Condition?patient= A Condition searchset Bundle: the patient's problems / diagnoses.
GET /fhir/AllergyIntolerance?patient= An AllergyIntolerance searchset Bundle: recorded allergies.
GET /fhir/MedicationRequest?patient= A MedicationRequest searchset Bundle, including e-prescribed medications.
Secured by the same model as the rest of the platform. Every FHIR endpoint is gated by the ehr.read permission and is tenant-scoped: a caller only ever sees their own tenant's data, enforced at the data-access layer. A missing permission returns HTTP 403.

Illustrative shape of a searchset Bundle

{
  "resourceType": "Bundle",
  "type": "searchset",
  "total": 2,
  "entry": [
    {
      "resource": {
        "resourceType": "Observation",
        "status": "final",
        "code": { "text": "Heart rate" },
        "valueQuantity": { "value": 72, "unit": "beats/min" }
      }
    }
  ]
}

Illustrative only: structure shown to convey shape, not a live response.

HL7 v2

HL7 v2 (the language your existing systems already speak)

HL7 v2 is the messaging standard that has run hospital integrations for decades. Almost every incumbent HIS, LIS and RIS already emits and consumes it. Greyquill handles it in both directions, so your current feeds keep working.

↓ Inbound: ADT drives patient state

POST /hl7/v2/ingest

Greyquill ingests inbound HL7 v2 messages and parses ADT events from the MSH/PID segments: A01 admit A03 discharge A08 update. Your existing ADT feed can drive patient state inside Greyquill directly.

No custom code to connect your ADT feed: point it at the ingest endpoint and admissions, discharges and updates flow straight through.

↑ Outbound: orders & results flow out

GET /hl7/v2/orders/:serviceRequestId/message

Greyquill generates outbound ORM / ORU messages for a service request, so orders and results can flow out to an existing LIS or RIS in standard HL7 v2.

The systems that fulfil orders and report results keep receiving them in the format they already understand.

Terminology

Terminology service

Coded data only stays useful if it stays consistent. Greyquill exposes FHIR terminology operations so diagnoses and other coded values are looked up, expanded and validated against real code systems, instead of drifting into free text.

OperationWhat it does
GET /terminology/CodeSystem Lists the available code systems the platform recognises.
GET /terminology/ValueSet/:id/$expand Expands a value set into its full list of permitted codes, e.g. to populate a picker.
GET /terminology/$lookup Looks up the details of a single code (its display, its system).
POST /terminology/$validate-code Validates that a code is legitimate for a given system or value set before it is stored.
Coded data done right: lookups and validation keep diagnoses and other coded fields consistent across departments and across the systems you exchange with.
Imaging

Where imaging interoperability fits

DICOM / PACS image transport is on the roadmap, not shipped. Greyquill's imaging modules cover the radiographer worklist and image capture / QC today, and clinical imaging data can travel as FHIR resources. Direct DICOM / PACS image transport, where this section will sit, is a phased item and is not yet available. We state this plainly so there is no surprise during integration planning.

How this enables coexistence

Because Greyquill both speaks FHIR R4 and exchanges HL7 v2, it does not have to be all or nothing. Greyquill can be the system of record for some departments while exchanging FHIR and HL7 with your incumbents for others. An existing ADT feed drives patient state inbound, orders and results flow outbound to your LIS / RIS, and FHIR exposes the record to anything else that needs it.
