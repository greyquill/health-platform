Architecture & trust

Secure, auditable, multi-tenant by design

The platform was built the way a hospital IT and security team would want to inspect it: every endpoint is permission-guarded, every query is scoped to its tenant, and every clinical write is recorded to an immutable audit trail. This page is for your CIO and security reviewer.

Architecture

One logical backend, one frontend codebase

A modern, modular stack: a single Fastify API serving every persona, a Flutter app that ships to web and mobile from one codebase, and a FHIR-aligned data model underneath.

A API layer

Fastify on Node.js, written end to end in TypeScript. Modular, per-feature design: each capability is its own slice, all composed into one logical backend and one deployment.

D Data layer

Drizzle ORM over PostgreSQL. A ~55-table, FHIR R4-aligned data model is the single source of truth for clinical, revenue, pharmacy, lab, imaging, facilities, EMS and compliance data.

F Client layer

Flutter staff app and patient app: web and mobile delivered from one shared codebase, so behaviour stays consistent across desktop, tablet and phone.

How a request flows

Every staff request passes the same gates before it ever touches data, and clinical writes are recorded after the response is sent.

Client (Flutter)
   │  authenticated request  (staff: email + password · patient: phone + OTP)
   ▼
Fastify route
   ├─ permission guard       → missing permission?  → HTTP 403
   ├─ tenant scope           → every query: WHERE tenant_id = caller's tenant
   ▼
Drizzle ORM → PostgreSQL     → response returned to client
   │
   └─ onResponse hook (non-blocking) → successful staff write → audit_events
Access control

Role-based access control on every endpoint

Roles do not just hide buttons in the UI. The server enforces about 28 granular permissions, and every endpoint declares the permission it requires. If the caller's account does not hold that permission, the request is rejected with HTTP 403, regardless of what the client app shows.

There are 13 staff roles plus a patient app, all behind one unified staff login: the role and its permissions come from the account, not from a separate set of credentials. A representative sample of roles and the permissions they carry:

RoleRepresentative permissionsPrimary worklist
Doctor ehr.read ehr.write orders.write rx.write Order entry, e-prescribe, clinical notes, results, decision support
Nurse vitals.write meds.administer notes.write eMAR / medication pass, vitals, nursing notes, SBAR handover
Head Nurse ward.manage vitals.write Ward board and bed management across wards
Pharmacist rx.dispense Dispense queue and pharmacy inventory
Lab Tech labs.write Lab result entry and specimen tracking
Radiographer imaging.read imaging.write Imaging modality worklist, image capture and QC
Biller billing.read billing.write AR worklist, payment posting, denials, reconciliation
Compliance Officer audit.read Read-only view over the audit / compliance trail
Admin Full permission set (admin KPI rollup) Admin dashboard: beds, AR invoices, KPI rollup

Other roles follow the same pattern: Facilities Tech (facilities.read/facilities.write), EMS Crew (ems.read/ems.write), Clinic Coordinator (clinic.read/clinic.write) and HR Manager (hr.read/hr.write), each scoped to exactly what its work requires.

Multi-tenancy

Hard data separation between facilities

One deployment can serve multiple facilities with no data bleed between them. Isolation is not a UI convenience: it is enforced where the data is read and written.

T Tenant-scoped queries

Every query is scoped to the caller's tenant (WHERE tenant_id = caller's tenant). A user in one tenant simply cannot retrieve another tenant's rows.

L Enforced at the data layer

The boundary lives at the data-access layer, not just in screens or filters. Even an unexpected request path is still scoped to its tenant before it reaches the database.

Why this matters for a group or network. A single Greyquill deployment can host several hospitals or sites, each with its own users, data and audit history, isolated from one another while sharing the same maintained codebase and upgrade path.
Audit & compliance

Every clinical write is on the record

The platform records who changed state, not who looked. A non-blocking onResponse hook captures every successful staff write into an audit_events table.

W What is captured

Successful staff writes on /staff routes: POST, PUT, PATCH and DELETE with a 2xx/3xx response. Each entry records the acting user, the patient (when patient-scoped) and the action.

R What is not

Reads are not logged, by design: the trail answers "who changed state," not "who looked at a screen." That keeps the record focused on the events that actually matter for accountability.

C Who reads it

The Compliance Officer role has read-only access over the audit trail and nothing else, so the people who watch the system are themselves scoped and accountable.

Auditing can never harm a clinical action. The hook runs after the response has already been sent and swallows its own errors. Recording an event cannot slow down, block, or fail a clinical write: the audit trail is a passive observer, never on the critical path.
Deployment & operations

Release safety the hospital can rely on

A native deployment with no Docker, predictable database migrations, and an automated per-persona check that gates every release.

N Native deploy

The Fastify API runs under pm2 or systemd against PostgreSQL, behind a reverse proxy, with a Cloudflare tunnel providing TLS and the public hostname. No container runtime to operate.

M Tracked migrations

Database changes are idempotent and tracked: a _migrations table records each .sql so it applies exactly once, transactionally. A migration either lands cleanly or rolls back.

S Smoke gate

A per-persona smoke check runs on every release as an automated go / no-go gate, before the change is considered live.

What the release smoke check actually does

Logs in as each role

Against the running server, the check authenticates as every staff role in turn, exactly as a real user would.

Exercises each role's worklist

For each role, it drives that role's primary worklist to confirm the screens that role depends on are working on the freshly deployed build.

Runs a cross-persona 403 check

It then confirms a role is refused access to another role's protected endpoint: proof that permission boundaries are still intact after the release.

The benefit to you. No release reaches your users without an automated pass that every persona can do its job and that the access controls still hold. Security regressions are caught by the gate, not by a patient.
Transparency

Where we are honest

This is a live pilot build with sample data. It runs end to end and is feature-complete across the modules shown on this site, but it is a demo / pilot environment with seeded sample data and demo credentials.

It is not yet formally certified. We make no HIPAA, ABDM, NABH or ISO certification claims. The compliance posture, role-based access control, full audit trail and hard tenant isolation, is built in and ready to be taken through certification together with a launch customer.

Some things are on the roadmap, not shipped. Real single sign-on (SSO), production patient-auth hardening, and DICOM / PACS image transport are planned and phased, not done. A few integrations (video consult, ambient scribe) are interface stubs today. We would rather you know exactly what is real now and what is coming.
